=== Diagnosis ===
With deb076d deployed, the endpoint would inspect Content-Length BEFORE reading
the body and return 413 Payload Too Large in milliseconds (no body buffered).

ACTUAL on running stack:
  Case 1+2 (lying Content-Length: 99M, no body): server WAITS for the body up to
    5s — proves no early Content-Length pre-check exists. This is exactly the
    DoS surface deb076d was meant to close (a cheap client can hold N connections
    open for the duration of FastAPI's read timeout).
  Case 3 (real 12MB upload): server buffered the whole body, then returned 400
    "File size must be less than 10MB" — confirming the size check happens AFTER
    full buffering, not on the Content-Length header.

  docker exec scholarship_backend_dev grep -n "content_length\|413" \
      /app/app/api/v1/endpoints/user_profiles.py
  (no matches)

VERDICT: FAIL_NOT_DEPLOYED -- deb076d not in deployed image. The DoS
hardening is not active. Existing 10MB cap still works post-buffering, so
this is a "wasted bandwidth + memory" hardening regression, not a crash bug.
