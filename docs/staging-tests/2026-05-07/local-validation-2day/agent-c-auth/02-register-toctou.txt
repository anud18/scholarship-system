=== Test methodology ===
After clearing the register rate-limit key in Redis, fire 2-3 concurrent
POST /api/v1/auth/register calls with the same nycu_id+email and verify
the loser(s) get HTTP 409, not 500. Repeated across 5 rounds (15 attempts).

round=1 req=2 HTTP=409 -> {"success":false,"message":"NYCU ID already exists","error_code":"CONFLICT"}
round=1 req=3 HTTP=409 -> {"success":false,"message":"NYCU ID already exists","error_code":"CONFLICT"}

=== Race coverage ===
9 winning attempts, 6 losing IntegrityError races -> all 6 losers returned clean 409.
Both branches of the IntegrityError handler were exercised: 5 hit the
"NYCU ID already exists" branch, 1 hit the "Email already registered" branch.

=== Anomaly: a single 500 was observed in the very first attempt of this session ===
trace_id=abfbd2b0-2df1-4aaf-827e-d6a58e945854

The full stack trace from backend logs shows the IntegrityError raising from
auth_service.py:54 (await self.db.commit()) and propagating up through
Starlette middleware as "Unhandled exception". The except IntegrityError
block at line 57 should have caught it. After 15 follow-up concurrent attempts
no 500 was reproducible.

Speculative root causes:
1. The first concurrent test happened during the very first warm-up of the
   asyncpg connection pool — a connection state issue may have prevented the
   except block from running cleanly.
2. The asyncpg-translated IntegrityError (sqlalchemy.dialects.postgresql.asyncpg.IntegrityError)
   may inherit from sqlalchemy.exc.IntegrityError differently in some edge case.

The fix (commits 0d06085/2e540b4/62de4e8) is mostly working but the single
500 suggests an edge case still exists. Worth filing as a separate issue
"intermittent IntegrityError leak in register_user during cold pool" if
reproducible from a backend restart.

=== Verdict ===
PASS_WITH_ANOMALY: 14/15 attempts handled cleanly. 1/15 leaked a 500 on
session-cold start. Recommend a follow-up issue to capture the cold-start
edge case.
